Mapingo Troubleshooting

인증, 캐시, 검증 경계를 분리한 미니프로젝트 이슈

Mapingo에서 다룬 문제는 서비스 API의 기본기가 드러나는 사례입니다. 인증 실패가 어디서 처리되어야 하는지, 캐시 무효화가 트랜잭션과 어떤 순서로 맞물려야 하는지, DTO 검증 규칙이 타입과 맞는지 확인했습니다.

Spring Security / JWT

JWT 인증 필터 예외가 500으로 반환되는 문제

만료되었거나 위변조된 토큰으로 보호된 API를 호출했을 때, 클라이언트가 기대한 401이 아니라 500 응답이 반환되어 재로그인 안내 흐름이 깨졌습니다.

Problem

인증 실패와 서버 내부 오류가 같은 500 계열처럼 보이면서 프론트엔드가 로그인 만료와 실제 장애를 구분할 수 없었습니다.

Cause

JWT 검증은 DispatcherServlet 이전의 Security Filter Chain에서 실행되므로, 필터 내부 토큰 만료/서명 불일치 예외는 Controller Advice가 잡지 못했습니다.

Solution

토큰 검증 실패를 AuthenticationException 계열로 정리하고 AuthenticationEntryPoint에서 401 상태 코드와 안내 메시지를 명시적으로 반환하도록 분리했습니다.

Result

토큰 누락, 만료, 위변조 요청을 재현해 모두 401로 응답되는지 확인했고, 정상 토큰 요청은 기존 API 흐름을 유지했습니다.

Redis / MySQL

Redis Cache-Aside 도입 후 데이터 정합성 문제

자주 조회되는 데이터를 Redis에 캐싱한 뒤, DB 값이 변경되었는데도 일부 요청에서 이전 데이터가 조회되는 문제가 있었습니다.

Problem

수정 요청 직후 새로고침하거나 다른 탭에서 조회하면 최신 DB 값과 캐시 값이 어긋날 수 있었습니다.

Cause

TTL 문제가 아니라 DB update, transaction commit, Redis delete 순서의 문제였습니다. 캐시 삭제가 커밋보다 먼저 실행되면 다른 요청이 오래된 값을 다시 캐시에 만들 수 있었습니다.

Solution

수정 로직 내부에서 바로 Redis를 삭제하지 않고, 트랜잭션 커밋 이후 캐시를 무효화하도록 이벤트 처리 흐름을 조정했습니다.

Result

수정 직후 단건 조회, 목록 조회, 반복 조회를 순서대로 실행하고 SQL 로그와 캐시 삭제 로그를 함께 비교해 커밋 이후 무효화가 실행되는지 확인했습니다.

Validation / Swagger

회원가입 요청 시 LocalDate 검증이 실패하는 문제

Swagger에서 회원가입 요청을 보낼 때 birthDate 필드 검증 중 No validator could be found for constraint 'NotBlank' validating type 'LocalDate' 예외가 발생했습니다.

Problem

사용자는 정상 입력처럼 보이는 요청을 보냈지만, 서버에서는 검증 설정 오류로 500 계열처럼 보일 수 있는 예외가 기록되었습니다.

Cause

@NotBlank는 String 전용 검증인데 LocalDate 타입인 birthDate에 적용되어 있었습니다. Swagger 예시와 실제 검증 규칙도 함께 정리되지 않아 반복될 여지가 있었습니다.

Solution

회원가입 DTO를 필드 단위로 다시 확인했습니다. String 필드는 공백과 형식 검증을 분리하고, LocalDate인 birthDate는 문자열 전용인 @NotBlank 대신 타입에 맞는 검증으로 교체했습니다.

비밀번호는 짧은 값이 통과하지 않도록 패턴 검증을 추가했고, 주소는 실제 입력에 필요한 숫자와 특수문자를 허용하도록 정규식을 조정했습니다. 소셜 회원가입에서는 기존 회원인지 판단하는 isNewUser 값을 로그인 응답까지 전달해 중복 가입 흐름도 함께 막았습니다.

Result

Swagger example 값으로 회원가입하면 500이 나던 이슈 #36과, 같은 원인의 birthDate 검증 오류인 이슈 #102는 타입 불일치가 원인임을 확인하고 PR #99에서 생일 검증 어노테이션을 수정했습니다.

이슈 #82처럼 비밀번호 1234도 가입되는 검증 누락은 PR #87에서 회원가입 유효성 검증 어노테이션을 추가해 보완했습니다. 이후 PR #107에서는 주소 정규식과 소셜 중복 가입 방지 로직까지 이어서 정리해, 일반 회원가입과 소셜 회원가입 모두 같은 검증 기준으로 처리되도록 맞췄습니다.
