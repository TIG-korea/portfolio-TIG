AI Assisted Workflow

AI를 설계, QA, 보안 검증에 붙여 사용한 기록

노션 기록을 바탕으로 AI를 요구사항 정리, 구현 명세, 공격 관점 QA에 활용한 과정을 정리했습니다. 생성 결과는 코드·데이터 범위·테스트 기준과 대조해 채택 여부를 결정했습니다.

Principles

AI 활용 기준

기능 목적을 먼저 고정

AI에게 바로 코드를 시키기 전에 사용자가 얻어야 하는 결과, 화면 역할, 데이터 범위를 먼저 정리했습니다.

응답 계약까지 함께 설계

프롬프트 문장만 바꾸지 않고 AI 서버 모델, Spring DTO, 프론트 타입과 UI까지 같이 바꿔 실제 화면에서 구분되게 했습니다.

검증을 코드로 남김

프롬프트 공격, 가이드 키 변조, JSON 실패, 운영 설정 위험처럼 재발 가능한 문제는 테스트와 PR 기록으로 남겼습니다.

요구사항 정의 / 구현 명세

게임가이드 AI 기능을 구현 가능한 명세로 압축

처음에는 패치노트 요약이나 일반 챗봇처럼 AI를 붙이는 방향도 고민했지만, 게임가이드의 진짜 문제는 정보가 부족한 것이 아니라 챔피언, 시너지, 아이템, 증강체 정보가 분리되어 사용자가 직접 연결해야 한다는 점이었습니다.

기능 방향

AI가 최강 덱을 만들어내는 기능이 아니라, 사용자가 선택한 가이드 항목을 하나의 운영 흐름으로 엮어주는 AI Pathfinder 역할로 범위를 좁혔습니다.

역할 범위

프론트는 선택 UI와 결과 패널을 만들고, Spring은 AI 서버 프록시를 담당하며, AI 서버는 선택된 가이드 데이터 안에서 요약, 운영 방향, 추천 탐색 항목, 주의점을 생성하도록 나눴습니다.

제약 조건

승률, 평균 등수, TOP4율처럼 현재 게임가이드 데이터에 없는 수치는 만들지 않는다는 제약을 명세 단계에서 먼저 넣었습니다.

Security / Cost / Reliability

AI 호출 비용과 응답 신뢰성을 계층별로 보호

React→Spring Boot→FastAPI→OpenAI 요청 경로를 완성한 뒤, 신규 API가 기존 인증과 호출 제한을 일부 우회할 수 있음을 리뷰에서 발견했습니다. 익명·반복 호출을 차단하고 사용자 입력과 모델 출력을 모두 서버에서 검증했습니다.

Risk

익명 사용자의 반복 호출은 비용 증가로, 존재하지 않는 가이드 키는 잘못된 추천 화면 이동으로 이어질 수 있었습니다.

Auth / Cost

Spring은 비로그인 요청을 401로 거부하고 로그인 사용자별 60초당 10회를 넘기면 429를 반환합니다. FastAPI는 OpenAI 호출 전에 프로세스별 60초당 30회를 제한하며, 이 전역 제한과 AI 장애·응답 파싱 실패는 Spring이 규칙 기반 대체 응답으로 전환합니다.

데이터 검증

Spring에서 요청의 패치 버전과 가이드 키를 DB와 대조했습니다. 모델이 반환한 키도 서버의 허용 목록으로 다시 확인해 존재하지 않는 항목을 응답과 화면 이동에서 제거했습니다.

Failure

응답 형식 검증, 공격 문구 검사, 장애 차단기, 규칙 기반 대체 응답을 조합했습니다. OpenAI 장애가 60초 안에 5회 누적되면 30초간 추가 호출을 막아 연쇄 실패를 줄였습니다.

Result

PR #643에서 프론트·Spring·AI 서버 테스트와 CI를 통과했습니다. 공격 입력과 출력 변조는 PR의 회귀 테스트로, QA 이슈 #676에서는 비로그인 401과 사용자별 제한 429 표시를 점검했습니다.

Limit

호출 제한과 장애 차단 상태는 프로세스 메모리에 저장됩니다. 다중 워커·인스턴스로 확장할 때는 Redis 등 공용 저장소로 옮겨야 합니다.

Prompt / Response Contract

가이드 근거와 AI 제안을 분리한 응답 구조

초기 프롬프트는 없는 수치를 만들지 않는 데 집중했지만, 실제 사용자는 운영 판단, 전환, 아이템 활용처럼 더 실전적인 설명을 기대했습니다. 그래서 답변을 길게 늘리는 대신 실제 데이터 근거와 AI의 참고 제안을 분리하는 구조로 바꿨습니다.

evidence_notes 실제 게임가이드 데이터에서 확인 가능한 근거만 담습니다.

creative_suggestions 데이터 밖의 확정 정보가 아니라, 상황에 따라 시도해볼 수 있는 운영 아이디어만 담습니다.

코드 반영 AI 서버 응답 모델, 대체 응답, Spring DTO, 프론트 API 타입, 챗봇 결과 UI까지 함께 수정했습니다.

공격 관점 QA / Prompt Injection Defense

사용자 입력과 AI 출력을 서버에서 이중 검증

시스템 프롬프트 탈취, API 키 유도, 허위 통계 생성, 가이드 키 조작 등 공격 시나리오를 직접 입력해 방어 로직을 검증했습니다.

Risk

사용자 질문, 대화 이력, 가이드 데이터는 모두 신뢰할 수 없는 입력이므로 AI가 이를 운영자 명령처럼 해석할 수 있습니다.

Defense

입력 정규화 후 알려진 공격 문구를 먼저 차단했습니다. 응답에서는 시스템 프롬프트·API 키 등 위험 문구와 허용되지 않은 가이드 키를 다시 검사했습니다.

Result

허용된 selected_entries, candidate_refs 밖의 가이드 키는 제거하고 이름이 변조되면 서버 기준 명칭으로 보정했습니다. 공격·환각 유도 사례와 재현 가능한 회귀 테스트를 구분해 확인했습니다.

AI 기반 요구사항 인터뷰 / UX QA

크롤링 결과를 그대로 보여주지 않고 탐색 가능한 패널로 정리

패치노트 페이지 QA에서는 왼쪽 요약 패널에 유닛: 1단계, 특성 같은 가공되지 않은 하이라이트가 노출되어 맥락이 약했습니다. AI와의 요구사항 인터뷰로 사용자가 이 영역에서 기대하는 정보를 다시 정리했습니다.

개선 전 크롤링된 하이라이트 문구가 리스트처럼 노출되어 사용자가 패치 흐름을 빠르게 파악하기 어려웠습니다.

개선 후 패치 요약과 정규화된 주요 변경 포인트만 남기고, 카테고리 이동은 기존 우측 탭에 맡겨 중복된 빠른 보기 버튼을 제거했습니다.

의미 AI를 화면 문구 생성에만 쓰지 않고, 사용자가 데이터를 어떤 단위로 이해해야 하는지 정리하는 QA 도구로 활용했습니다.

AI 기반 배포 QA / Release Guard

운영 배포에 로컬 테스트 데이터가 섞일 위험 차단

기본 docker-compose.yml의 Flyway 경로에 로컬 테스트용 초기 데이터가 포함될 수 있었습니다. 그대로 배포하면 샘플 데이터나 로컬 관리자 계정이 운영 DB에 들어갈 수 있는 문제였습니다.

Risk

기본 Compose 실행에서 정식 마이그레이션과 로컬 테스트 초기 데이터가 함께 적용될 수 있었습니다.

Fix

기본 Compose는 classpath:db/migration만 사용하고, 로컬 배포 확인이 필요할 때만 docker-compose.local-smoke.yml 덮어쓰기 설정을 붙이도록 분리했습니다.

Result

이슈 #650에 위험을 기록하고 PR #651에서 운영용 마이그레이션과 로컬 테스트 데이터를 분리했습니다. 운영 기본 경로에서는 테스트 데이터가 적용되지 않고, 로컬 QA에서는 별도 설정으로 같은 데이터를 재현하도록 검증했습니다.
