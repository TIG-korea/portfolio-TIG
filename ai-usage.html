AI Assisted Workflow

AI를 설계, QA, 보안 검증에 붙여 사용한 기록

노션의 AI 활용 로그는 단순히 코드 생성을 맡긴 기록이 아니라, 요구사항을 구조화하고, 응답 계약을 설계하고, 프롬프트 공격과 배포 위험을 점검한 기록입니다. AI가 낸 결과를 그대로 믿기보다 프로젝트 코드, 데이터 범위, 테스트 기준에 맞게 검증하는 흐름으로 정리했습니다.

Principles

AI 활용 기준

기능 목적을 먼저 고정

AI에게 바로 코드를 시키기 전에 사용자가 얻어야 하는 결과, 화면 역할, 데이터 범위를 먼저 정리했습니다.

응답 계약까지 함께 설계

프롬프트 문장만 바꾸지 않고 AI 서버 모델, Spring DTO, 프론트 타입과 UI까지 같이 바꿔 실제 화면에서 구분되게 했습니다.

검증을 코드로 남김

프롬프트 공격, ref 변조, JSON 실패, 운영 설정 위험처럼 재발 가능한 문제는 테스트와 PR 기록으로 남겼습니다.

ooo seed / Feature Spec

게임가이드 AI 기능을 구현 가능한 명세로 압축

처음에는 패치노트 요약이나 일반 챗봇처럼 AI를 붙이는 방향도 고민했지만, 게임가이드의 진짜 문제는 정보가 부족한 것이 아니라 챔피언, 시너지, 아이템, 증강체 정보가 분리되어 사용자가 직접 연결해야 한다는 점이었습니다.

Direction

AI가 최강 덱을 만들어내는 기능이 아니라, 사용자가 선택한 가이드 항목을 하나의 운영 흐름으로 엮어주는 AI Pathfinder 역할로 범위를 좁혔습니다.

Scope

프론트는 선택 UI와 결과 패널을 만들고, Spring은 AI 서버 프록시를 담당하며, AI 서버는 선택된 가이드 데이터 안에서 요약, 운영 방향, 추천 탐색 항목, 주의점을 생성하도록 나눴습니다.

Guard

승률, 평균 등수, TOP4율처럼 현재 게임가이드 데이터에 없는 수치는 만들지 않는다는 제약을 명세 단계에서 먼저 넣었습니다.

Prompt / Response Contract

가이드 근거와 AI 제안을 분리한 응답 구조

초기 프롬프트는 없는 수치를 만들지 않는 데 집중했지만, 실제 사용자는 운영 판단, 전환, 아이템 활용처럼 더 실전적인 설명을 기대했습니다. 그래서 답변을 길게 늘리는 대신 실제 데이터 근거와 AI의 참고 제안을 분리하는 구조로 바꿨습니다.

evidence_notes 실제 게임가이드 데이터에서 확인 가능한 근거만 담습니다.

creative_suggestions 데이터 밖의 확정 정보가 아니라, 상황에 따라 시도해볼 수 있는 운영 아이디어만 담습니다.

코드 반영 AI 서버 응답 모델, fallback 응답, Spring DTO, 프론트 API 타입, 챗봇 결과 UI까지 함께 수정했습니다.

ooo qa / Prompt Attack Defense

프롬프트 공격을 사용자 질문으로 보고 서버에서 한 번 더 방어

노션에서는 개선한 방어 로직을 공격자 입장에서 다시 검증했습니다. 시스템 프롬프트 탈취, API 키 유도, 이전 지시 무시, 정상 질문에 공격 섞기, JSON 형식 깨기, 없는 통계 생성, ref 조작 같은 입력을 직접 넣어봤습니다.

Risk

사용자 질문, 대화 이력, 가이드 데이터는 모두 신뢰할 수 없는 입력이므로 AI가 이를 운영자 명령처럼 해석할 수 있습니다.

Defense

프롬프트에 방어 규칙을 추가하고, 줄바꿈/따옴표/백틱을 정리해 명령문처럼 구조화되지 않게 만들었습니다. 응답 후에는 system prompt, OPENAI_API_KEY, internal secret 같은 위험 문구를 검사해 fallback으로 전환했습니다.

Result

허용된 selected_entries, candidate_refs 밖의 ref는 제거하고, 이름이 변조되면 서버가 원래 가이드명으로 보정했습니다. AI 서버 전체 테스트는 66 passed까지 확인했습니다.

ooo interview / UX QA

크롤링 결과를 그대로 보여주지 않고 탐색 가능한 패널로 정리

패치노트 페이지 QA에서는 왼쪽 요약 패널에 유닛: 1단계, 특성 같은 raw 하이라이트가 그대로 노출되어 맥락이 약한 문제가 있었습니다. AI interview 방식으로 사용자가 이 영역에서 기대하는 정보를 다시 정리했습니다.

Before 크롤링된 하이라이트 문구가 리스트처럼 노출되어 사용자가 패치 흐름을 빠르게 파악하기 어려웠습니다.

After 패치 요약 문장, 정리된 주요 변경 포인트, 카테고리별 빠른 보기 버튼으로 바꿨습니다.

Meaning AI를 화면 문구 생성에만 쓰지 않고, 사용자가 데이터를 어떤 단위로 이해해야 하는지 정리하는 QA 도구로 활용했습니다.

ooo qa / Release Guard

운영 배포 설정에서 local-smoke seed 혼입 위험 발견

운영 배포 설정을 QA하던 중 기본 docker-compose.yml의 Flyway 경로에 로컬 테스트용 db/local-smoke seed가 포함될 수 있는 위험을 찾았습니다. 운영 DB에 샘플 데이터나 로컬 관리자 계정이 들어갈 수 있는 설정 문제였습니다.

Risk

기본 compose 실행에서 정식 migration과 local-smoke seed가 함께 적용될 수 있었습니다.

Fix

기본 compose는 classpath:db/migration만 사용하고, 로컬 스모크가 필요할 때만 docker-compose.local-smoke.yml override를 명시적으로 붙이도록 분리했습니다.

Result

이슈 #650에는 기본 배포 compose에서 local-smoke seed가 함께 실행될 수 있는 운영 위험을 기록했습니다. 이 설정이 그대로 배포되면 샘플 데이터나 로컬 관리자 계정이 운영 DB에 들어갈 수 있었기 때문에, 단순 문서 수정이 아니라 배포 기본값을 바꾸는 문제로 분리했습니다.

PR #651에서는 기본 docker-compose.yml의 Flyway 경로를 정식 migration만 보도록 정리하고, local-smoke seed는 docker-compose.local-smoke.yml override를 명시적으로 붙일 때만 실행되도록 바꿨습니다.

그 결과 운영 배포 기본 경로에서는 local-smoke seed가 자동 적용되지 않고, 로컬 QA가 필요할 때만 같은 seed를 재현할 수 있게 됐습니다. 이 분리 기준으로 QA 결과를 다시 확인해 PASS로 남겼습니다.
