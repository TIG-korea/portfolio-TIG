TFT-gogo Troubleshooting

성능, 데이터, 배포 리스크를 실제 수치로 확인한 파이널 프로젝트 이슈

TFT-gogo에서는 게임 가이드와 패치노트 도메인을 맡으면서 외부 데이터 import, API 응답 성능, DB migration, 화면 분류 품질을 함께 다뤘습니다. Notion 로그에서 포트폴리오에 어필하기 좋은 문제만 골라 정리했습니다.

Performance / API Flow

게임가이드 카탈로그 선호출 성능 개선 분석

첫 진입에 필요한 정보는 현재 패치 버전 정도였지만, 프론트엔드는 챔피언, 시너지, 아이템, 증강체 전체 카탈로그를 반환하는 /api/guide를 먼저 호출하고 있었습니다.

Problem

장애 수준은 아니었지만 게임가이드 API 중 /api/guide가 가장 무거운 호출로 측정됐고, 사용자 첫 진입 흐름에 불필요한 대용량 조회가 포함되어 있었습니다.

Cause

현재 패치 버전만 필요할 때도 전체 카탈로그 데이터를 조회하고 응답으로 내려 DB 조회량, 응답 생성 비용, 네트워크 전송량이 함께 커졌습니다.

Solution

첫 진입은 /api/guide/patch-version으로 가볍게 처리하고, 탭 접근 시 /api/guide/traits, /api/guide/items, /api/guide/augments, /api/guide/champions를 분리 조회하는 방향으로 정리했습니다.

JMeter / Query Optimization

패치노트 changes 조회 통계 쿼리 중복 개선

실제 Riot TFT 17.6 패치노트를 import한 로컬 DB 기준으로 공개 패치노트 API 부하 테스트를 진행했습니다. 에러율은 0%였지만 GET /api/patch-notes/{version}/changes 계열이 목록 조회보다 지속적으로 무거웠고, 변경 전 최대 응답시간은 9,696ms까지 튀었습니다.

Problem

changes 조회가 목록 조회보다 평균, p95, p99가 높게 나타났고 통계 계산 때문에 DB round-trip이 반복되었습니다.

Cause

buildStats()가 category별 count, type별 count, high impact count를 각각 repository count 쿼리로 호출하고 있었습니다.

Solution

category, changeType, impact를 한 번에 group by 하는 projection 쿼리를 추가하고, 기존 응답 스키마는 유지한 채 통계 계산만 단일 결과 순회 방식으로 바꿨습니다.

Result

PR #701에서는 최근 패치노트 목록과 changes 조회 흐름을 함께 정리하면서 통계 쿼리를 단일 group by 조회로 바꿨습니다. 수정 후 PatchNoteServiceImplTest와 백엔드 전체 테스트를 돌리고, 로컬 실제 데이터 API smoke까지 확인했습니다. 이슈 #710에는 JMeter 50 users, 300 seconds, 3회 측정 결과를 댓글로 남겨 개선 전후 수치를 추적할 수 있게 했습니다.

Flyway / Release Smoke

기존 migration 수정으로 인한 백엔드 배포 실패 리스크

패치노트 이력 조회 성능을 위해 인덱스를 추가하면서 새 migration뿐 아니라 이미 존재하던 V1__init_schema.sql에도 같은 인덱스를 추가했고, 기존 DB에서 Flyway checksum mismatch로 backend startup이 중단됐습니다.

Problem

Docker MySQL + backend 컨테이너 스모크에서 Flyway validation 실패로 /actuator/health까지 도달하지 못했습니다.

Cause

Flyway의 V*.sql은 한 번 적용되면 checksum으로 추적되므로, 이미 적용됐을 수 있는 V1을 최신 스키마 스냅샷처럼 수정하면 기존 DB가 시작되지 않습니다.

Solution

V1 수정분을 제거하고, 인덱스 추가는 새 forward migration인 V21__add_patch_notes_history_index.sql에서만 수행하도록 분리했습니다. 같은 실수를 막기 위해 spec에 Flyway migration guardrail도 추가했습니다.

Result

깨끗한 새 MySQL 볼륨에서 V1부터 V21까지 정상 적용됐고, flyway_schema_history의 V21 success=1, idx_patch_notes_history 생성, backend health UP, /api/patch-notes 응답을 확인했습니다.

Patch Note Import / Data Quality

패치노트 자동 분류 보강: 챔피언/시너지 이름 기반 매칭

Riot 패치노트 원문은 항상 챔피언, 시너지 같은 상위 제목 구조를 안정적으로 제공하지 않았습니다. 그래서 트위스티드 페이트는 챔피언 변경인데 SYSTEM으로, 길잡이 관련 변경은 시너지 탭이 아니라 다른 탭으로 분류될 수 있었습니다.

Problem

API는 정상 응답하지만 백엔드에 저장되는 category 값이 틀려 프론트 화면의 탭 분류, 검색, 카운트가 함께 흔들리는 데이터 품질 문제였습니다.

Cause

기존 로직은 headingPath, sectionTitle, groupTitle의 키워드만 보고 카테고리를 정했고, row 안의 실제 대상 이름이 챔피언인지 시너지인지 확인하지 않았습니다.

Solution

같은 패치 버전의 게임가이드 데이터에서 챔피언 이름 목록과 시너지 이름 목록을 만들고, row 텍스트와 제목 경로에서 가장 긴 이름 매칭을 기준으로 CHAMPION 또는 TRAIT를 보강했습니다. 단, 버그 수정 row는 SYSTEM 우선 분류를 유지했습니다.

Result

트위스티드 페이트는 CHAMPION, 길잡이는 TRAIT로 분류되도록 테스트를 추가했고, AdminPatchNoteServiceImplTest, 백엔드 전체 테스트, PR CI를 통과했습니다.

Display QA

노션의 6.17, 6.18 QA 로그에서는 분류뿐 아니라 표시 품질도 함께 점검했습니다. 긴 원문 문장이 카드 밖으로 밀리거나 제목과 상세가 중복되는 문제를 줄이기 위해 (10) 같은 단계 표기를 제거하고, 콤마와 콜론 기준으로 제목과 상세 수치를 분리했습니다. 버그 수정 항목은 문장 안의 증가/감소 키워드 때문에 상향/하향처럼 보이지 않도록 SYSTEM + ADJUST 우선 규칙으로 정리했습니다.

ERD / JPA Entity

ERD와 백엔드 스키마 불일치 정리

패치노트/변경사항 API를 로컬 DB 기준으로 확인하던 중, 백엔드 코드가 팀 ERD와 다른 예전 컬럼과 테이블명을 바라보고 있었습니다. 대표적으로 patch_notes.is_active, patch_changes, image_url, description이 섞여 있었습니다.

Problem

기대 기준은 ERD의 patch_notes, patch_note_changes였지만, 실제 백엔드와 로컬 SQL 일부는 예전 스키마를 조회해 Unknown column 'pn1_0.is_active'가 발생했습니다.

Cause

팀 ERD는 deleted_at 기준으로 삭제 여부를 관리하고 변경사항 테이블도 patch_note_changes를 사용했지만, 엔티티, Repository, local-smoke SQL에 예전 구조가 남아 있었습니다.

Solution

JPA Entity와 Repository 쿼리, 로컬 DB 스키마를 ERD 기준으로 대조하고, patch_changes를 patch_note_changes로 맞추며 description은 content, image_url은 representative_image_url로 정리했습니다.

Result

기존 패치노트/변경사항 데이터는 유지하면서 로컬 MySQL과 백엔드 코드가 같은 ERD 기준을 보도록 맞췄고, 백엔드 테스트와 프론트 빌드/테스트까지 확인했습니다.
