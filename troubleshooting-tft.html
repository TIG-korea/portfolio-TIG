TFT-gogo Troubleshooting

성능, 데이터, 배포 리스크를 실제 수치로 확인한 파이널 프로젝트 이슈

TFT-gogo에서 게임 가이드와 패치노트 도메인을 맡아 외부 데이터 수집, API 응답 성능, DB 마이그레이션, 화면 분류 품질을 함께 다뤘습니다. 노션 기록에서 문제·원인·검증 근거가 분명한 사례만 골라 정리했습니다.

Performance / API Flow

게임가이드 조회 최적화: DB 페이지네이션 구현과 후속 개선 검증

GET /api/guide/{tab}은 page와 pageSize를 받았지만 전체 데이터를 메모리에 올린 뒤 서비스에서 잘라냈고, 첫 진입은 현재 패치 버전만 필요해도 전체 카탈로그를 선조회했습니다.

Problem

API 계약에는 페이지 필드가 있었지만 Repository는 전체 List를 반환해 데이터가 늘수록 DB I/O와 서버 메모리 사용량이 함께 증가하는 구조였습니다.

직접 구현

PR #608에서 챔피언·시너지·아이템·증강체 조회를 DB 페이지네이션으로 전환해 필요한 행만 가져오도록 했습니다. 검색·등급·유효 데이터 조건을 목록과 전체 건수 쿼리에 동일하게 적용해 기존 응답 형식을 유지했습니다.

직접 검증

목록과 전체 건수가 같은 필터 조건으로 계산되는지 서비스·저장소 테스트와 전체 백엔드 테스트로 검증했습니다.

Concurrency / Content Refresh

다중 인스턴스에서도 패치→가이드 갱신 순서를 보장

패치노트와 게임가이드가 각자 정상 갱신돼도 서로 다른 lock과 진입점을 사용하면, 서버 A의 패치 커밋 전에 서버 B가 이전 current 버전으로 가이드를 만들 수 있었습니다. 각 API는 성공하지만 사용자에게 보이는 두 콘텐츠 버전은 어긋나는 경쟁 조건입니다.

Problem

AtomicBoolean은 한 서버 안에서만 동작하고, 작업별 락만으로는 “패치 저장 후 같은 버전의 가이드를 갱신한다”는 실행 순서를 보장할 수 없었습니다.

Cause

패치와 가이드가 서로 다른 락을 사용했고, 가이드 작업이 DB의 current를 다시 조회해 패치 트랜잭션의 커밋 시점과 경쟁했습니다. 자동 스케줄러와 관리자 수동 수집도 진입점이 달랐습니다.

Solution

패치→가이드 전체 실행 구간을 공용 MySQL 락(GET_LOCK)으로 보호했습니다. 저장이 끝난 패치 버전을 가이드 작업에 직접 전달하고, 패치 실패나 빈 버전이면 가이드를 실행하지 않았습니다. 자동·수동 경로는 같은 보호 로직을 사용하며, 다른 갱신이 진행 중인 수동 요청에는 HTTP 409를 반환합니다.

Result

락 경합과 각 실패 경로를 테스트하고, 실제 MySQL 8의 두 연결에서 첫 GET_LOCK=1, 경쟁 연결 0, RELEASE_LOCK=1을 확인했습니다. 가이드 갱신이 실패하면 이미 저장된 패치는 되돌리지 않고 기존 활성 가이드를 보존해 다음 실행에서 재시도합니다.

직접 기여

경쟁 조건 분석, 락·트랜잭션 경계 설계, 자동·수동 실행 통합, 실패 복구 테스트와 실제 MySQL 경합 검증을 수행했습니다. 이슈 #748에서 문제를 정의하고 PR #773으로 develop에 반영했습니다.

Import / Data Safety

부분 파싱 결과가 기존 패치노트를 지우지 못하게 차단

외부 페이지가 HTTP 200을 반환하고 파서 오류가 없어도, 페이지 구조 변경이나 최대 행 제한으로 일부만 추출될 수 있습니다. 기존 로직은 이를 정상 수집으로 보고 새 목록에 없는 기존 자동 수집 행을 삭제 대상으로 판단할 수 있었습니다.

Problem

부분 파싱이 정상 커밋되면 공개 API는 200을 반환하면서도 기존 패치 변경사항이 조용히 대량 유실될 수 있었습니다. HTTP 성공·본문 파싱 성공과 운영 반영 가능한 데이터 완전성이 구분되지 않은 상태였습니다.

Counterexample

기존·신규 결과가 각각 100개여도 수집 식별자의 교집합이 0이면 기존 데이터가 전부 삭제될 수 있습니다. 단순 행 수 대신 실제 삭제 기준과 같은 수집 식별자로 안전성을 판단했습니다.

Solution

0건 결과, 치명적 max detail rows reached 경고, 필수값 누락, 중복 수집 식별자를 DB 변경 전에 거부했습니다. 기존·신규 자동 수집 식별자의 유지율이 50% 미만이면 PATCH_NOTE_INVALID_DATA로 중단하고, 수동 편집 행은 유지율 계산과 삭제 대상에서 제외했습니다.

Result

100→20으로 급감한 경우, 정확히 50%인 경계, 행 수는 같지만 식별자 교집합이 0인 경우를 검증했습니다. PR #753 시점의 집중 테스트 28개와 백엔드 전체 테스트 352개가 통과했습니다.

Meaning

실제 장애를 복구한 사례가 아니라, 외부 구조 변경이 기존 데이터를 대량 삭제할 수 있는 경로를 사전에 차단한 사례입니다. 수집 성공 기준을 “요청 성공”에서 “데이터 완전성 검증 통과”로 확장했습니다.

JMeter / Query Optimization

패치노트 changes 조회 통계 쿼리 중복 개선

실제 TFT 17.6 데이터를 넣은 로컬 백엔드(localhost:8081)와 MySQL 8 환경에서 변경 전후를 각각 3회 비교했습니다(가상 사용자 50명, 300초). 오류율은 0%였지만 GET /api/patch-notes/{version}/changes 계열의 최대 응답시간은 변경 전 9,696ms까지 늘어났습니다.

Problem

changes 조회가 목록 조회보다 평균, p95, p99가 높게 나타났고 통계 계산 때문에 DB round-trip이 반복되었습니다.

Cause

buildStats()가 category별 count, type별 count, high impact count를 각각 repository count 쿼리로 호출하고 있었습니다.

Solution

카테고리·변경 유형·영향도를 한 번에 집계하는 쿼리를 추가하고, 기존 응답 형식은 유지한 채 통계 계산을 한 번의 결과 처리로 바꿨습니다.

Result

PR #712에서 여러 건수 조회를 단일 집계 쿼리로 통합했습니다. 서비스 테스트와 백엔드 전체 테스트, 로컬 실데이터 API 확인을 거쳐 이슈 #710에 같은 조건의 변경 전후 측정값을 남겼습니다.

대안 비교

통계 계산을 프론트로 옮기는 대안도 별도의 일회성 고부하 근사 실험으로 탐색했지만, 반복 검증된 정량 성과로 채택하지 않았습니다.

1차 개선: 중복 count 쿼리 제거

같은 응답 형식을 유지하면서 여러 건수 조회를 단일 집계 쿼리로 묶었습니다. 위와 같은 조건의 4개 공개 API 혼합 시나리오에서 평균과 상위 지연시간을 비교했습니다. 당시 데이터 분포상 CHAMPION·NERF 필터 2개는 0건 응답이었으므로, 결과는 해당 혼합 조건 안에서만 해석했습니다.

2차 검증: 통계 계산을 어디서 할 것인가

처음에는 통계 계산을 프론트로 넘기면 백엔드 부하가 줄어들 것으로 예상했습니다. 별도의 일회성 고부하 근사 비교에서는 목록 조회와 DB 연결 대기가 후속 병목 후보로 나타났지만, 실제 브라우저 실험이나 반복 측정이 아니어서 상세 수치를 성과 근거에서 제외했습니다.

계산 위치를 바꿀 근거는 부족해 백엔드 집계를 유지하고, 후속 후보를 캐싱·API 분리·실행계획과 인덱스·DB 연결 풀 점검으로 좁혔습니다.

Flyway / Release Smoke

기존 마이그레이션 수정으로 발생한 배포 실패 해결

로컬 Docker Compose 배포 확인에서 이미 적용된 V1__init_schema.sql이 함께 수정된 사실을 발견했습니다. 기존 DB에 기록된 Flyway 체크섬과 파일이 달라 백엔드 시작이 중단됐습니다.

Problem

Docker MySQL + backend 컨테이너 스모크에서 Flyway validation 실패로 /actuator/health까지 도달하지 못했습니다.

Cause

Flyway의 V*.sql은 한 번 적용되면 checksum으로 추적되므로, 이미 적용됐을 수 있는 V1을 최신 스키마 스냅샷처럼 수정하면 기존 DB가 시작되지 않습니다.

Solution

V1 수정분을 되돌리고 인덱스는 새 순방향 마이그레이션인 V21__add_patch_notes_history_index.sql에서만 추가하도록 분리했습니다. 이후 이미 적용된 마이그레이션은 수정하지 않는 기준도 명세에 남겼습니다.

Result

새 MySQL 볼륨에서 V1부터 V21까지 정상 적용되는지, 인덱스 생성과 백엔드 상태, /api/patch-notes 응답까지 확인했습니다.

Patch Note Import / Data Quality

패치노트 자동 분류 보강: 챔피언/시너지 이름 기반 매칭

Riot 패치노트 원문은 항상 챔피언, 시너지 같은 상위 제목 구조를 안정적으로 제공하지 않았습니다. 그래서 트위스티드 페이트는 챔피언 변경인데 SYSTEM으로, 길잡이 관련 변경은 시너지 탭이 아니라 다른 탭으로 분류될 수 있었습니다.

Problem

API는 정상 응답하지만 백엔드에 저장되는 category 값이 틀려 프론트 화면의 탭 분류, 검색, 카운트가 함께 흔들리는 데이터 품질 문제였습니다.

Cause

기존 로직은 headingPath, sectionTitle, groupTitle의 키워드만 보고 카테고리를 정했고, row 안의 실제 대상 이름이 챔피언인지 시너지인지 확인하지 않았습니다.

Solution

같은 패치 버전의 게임가이드 데이터에서 챔피언 이름 목록과 시너지 이름 목록을 만들고, row 텍스트와 제목 경로에서 가장 긴 이름 매칭을 기준으로 CHAMPION 또는 TRAIT를 보강했습니다. 단, 버그 수정 row는 SYSTEM 우선 분류를 유지했습니다.

Result

트위스티드 페이트는 CHAMPION, 길잡이는 TRAIT로 분류되도록 테스트를 추가했고, AdminPatchNoteServiceImplTest, 백엔드 전체 테스트, PR CI를 통과했습니다.

Display QA

노션의 6.17, 6.18 QA 로그에서는 분류뿐 아니라 표시 품질도 함께 점검했습니다. 긴 원문 문장이 카드 밖으로 밀리거나 제목과 상세가 중복되는 문제를 줄이기 위해 (10) 같은 단계 표기를 제거하고, 콤마와 콜론 기준으로 제목과 상세 수치를 분리했습니다. 버그 수정 항목은 문장 안의 증가/감소 키워드 때문에 상향/하향처럼 보이지 않도록 SYSTEM + ADJUST 우선 규칙으로 정리했습니다.

ERD / JPA Entity

ERD와 백엔드 스키마 불일치 정리

패치노트/변경사항 API를 로컬 DB 기준으로 확인하던 중, 백엔드 코드가 팀 ERD와 다른 예전 컬럼과 테이블명을 바라보고 있었습니다. 대표적으로 patch_notes.is_active, patch_changes, image_url, description이 섞여 있었습니다.

Problem

기대 기준은 ERD의 patch_notes, patch_note_changes였지만, 실제 백엔드와 로컬 SQL 일부는 예전 스키마를 조회해 Unknown column 'pn1_0.is_active'가 발생했습니다.

Cause

팀 ERD는 deleted_at 기준으로 삭제 여부를 관리하고 변경사항 테이블도 patch_note_changes를 사용했지만, 엔티티·저장소·로컬 테스트 SQL에 예전 구조가 남아 있었습니다.

Solution

JPA Entity와 Repository 쿼리, 로컬 DB 스키마를 ERD 기준으로 대조하고, patch_changes를 patch_note_changes로 맞추며 description은 content, image_url은 representative_image_url로 정리했습니다.

Result

PR #433에서 기존 패치노트/변경사항 데이터는 유지하면서 로컬 MySQL과 백엔드 코드가 같은 ERD 기준을 보도록 맞췄고, 백엔드 테스트와 프론트 빌드/테스트까지 확인했습니다.
