Decision Records

주요 기술적 의사결정

Mapingo

JWT + Redis 인증 흐름을 단계적으로 고도화

배경

일반 로그인, Google OAuth2, 로그아웃, 토큰 재발급이 모두 같은 인증 경계를 공유해야 했습니다.

선택

Access Token은 요청 인증에 사용하고, Refresh Token은 Redis에 저장해 Rotation과 로그아웃 시 폐기를 관리했습니다. 이후 OAuth 응답은 HttpOnly Cookie 기반으로 전환했습니다.

이유

재발급과 로그아웃을 서버에서 통제할 수 있고, 토큰이 URL이나 응답 body에 직접 노출되는 위험을 줄일 수 있었습니다.

검증

Redis 저장값과 TTL, 재발급 전후 토큰 불일치, OAuth 1회용 코드 교환, HttpOnly Cookie 동작을 PR 캡처와 Swagger 흐름으로 확인했습니다.

Mapingo

DTO와 Swagger 문서를 구현 코드와 분리

배경

여러 팀원이 동시에 도메인을 개발하면서 요청/응답 예시와 실제 DTO가 어긋나는 문제가 생겼습니다.

선택

Controller 문서화 클래스를 분리하고 Request/Response DTO 네이밍, responseCode 예시, 입력 검증 기준을 맞췄습니다.

이유

프론트엔드와 QA가 Swagger를 기준으로 API를 확인하기 때문에 문서 불일치 자체가 기능 결함으로 이어질 수 있었습니다.

검증

회원가입 검증, 비밀번호 정책, birthDate 검증 오류를 이슈 단위로 재현하고 Swagger 응답과 테스트 결과를 남겼습니다.

Mapingo

PortOne 결제 검증을 서버 기준으로 처리

배경

유료 기능 진입 전에 결제 성공 여부와 구독 활성 상태를 신뢰할 수 있는 기준으로 확인해야 했습니다.

선택

PortOne V1 API의 imp_uid 기반 조회와 금액 검증을 서버에서 수행하고 payment/subscription 엔티티로 구독 상태를 관리했습니다.

이유

클라이언트 결과만 신뢰하지 않고 외부 결제 API의 실제 응답과 서버 상태를 맞춰야 결제 우회 가능성을 줄일 수 있었습니다.

검증

결제 조회, 금액 불일치, 구독 취소, 관리자 결제내역 조회 조건을 PR 본문과 Swagger 응답으로 확인했습니다.

TFT-gogo

게임 가이드 데이터를 패치 버전 기준으로 고정

배경

챔피언, 아이템, 시너지, 증강체 정보는 패치 버전에 따라 의미가 바뀌므로 화면과 API가 같은 기준을 바라봐야 했습니다.

선택

CDragon 데이터를 import하고, 게임 가이드와 패치노트 화면에서 패치 버전 기준 조회가 흔들리지 않도록 API 계약을 먼저 정리했습니다.

이유

외부 데이터가 갱신되는 시점과 사용자가 조회하는 시점을 분리하지 않으면 최신 데이터와 화면 설명이 어긋날 수 있었습니다.

검증

scheduler, import 결과, API 응답 필드, 패치 필터 동작을 PR과 화면 캡처로 확인했습니다.

TFT-gogo

패치노트 import를 source 기준으로 중복 방지

배경

공식 패치노트는 주기적으로 갱신되지만, 같은 문서를 반복 import하면 중복 데이터와 운영 부하가 생길 수 있었습니다.

선택

서버 시작 시 1회 import, 매시간 목록 확인, 매일 06:30 상세 refresh로 나누고 DB에 없는 sourceKey/sourceUrl/version만 상세 import하도록 설계했습니다.

이유

수집 실패 fallback, 중복 실행 방지, 최신 데이터 반영을 동시에 고려해야 했기 때문입니다.

검증

로컬/dev 기본값은 OFF로 두고 공용 개발 서버나 운영 서버에서만 켜도록 분리했습니다. 단일 서버 기준 중복 실행은 AtomicBoolean guard로 막고, 멀티 서버 환경은 공용 락 후속 이슈로 분리했습니다.

TFT-gogo

AI 기능은 FastAPI 서버와 Spring 프록시로 역할 분리

배경

GameGuide AI Pathfinder는 프론트 채팅 UI, Spring API, FastAPI AI 서버, OpenAI 응답이 함께 맞물리는 기능이었습니다.

선택

AI 생성은 FastAPI에서 처리하고, Spring은 인증/프록시/응답 DTO 경계, React는 선택 UI와 결과 패널을 담당하도록 나눴습니다.

이유

AI 응답 모델을 독립적으로 조정하면서도 기존 서비스 API 계약과 화면 상태를 안정적으로 유지할 수 있었습니다.

검증

응답을 evidence_notes와 creative_suggestions로 분리하고, 프롬프트 공격 문구와 허용되지 않은 ref 변조를 서버에서 검사했습니다. AI 서버 테스트는 66 passed까지 확인했습니다.

공통

운영 배포 설정은 로컬 QA 설정과 분리

배경

배포 환경에서는 CORS, OAuth redirect, AI 서버 URL, OpenAI 키, DB migration, healthcheck, ALB 라우팅처럼 실패 지점이 많았습니다.

선택

환경 변수와 secret 파일을 분리하고, Docker/Flyway 기본값에서 local-smoke seed가 섞이지 않도록 운영 기본 경로를 정리했습니다.

이유

로컬 검증용 데이터나 관리자 계정이 운영 DB에 들어가는 위험은 기능 오류보다 복구 비용이 큰 운영 사고로 이어질 수 있었습니다.

검증

배포 전 체크리스트와 smoke 기준으로 주요 화면, 게임 가이드 조회, 패치노트 조회, AI Pathfinder 요청, 로그와 환경 변수를 순서대로 확인했습니다.

Operating Rules

앞으로도 유지할 기준

Issue, Branch, PR 연결

구현 의도, 변경 범위, 확인 방법을 PR에 남기고 리뷰 코멘트가 코드에 어떻게 반영됐는지 추적 가능하게 관리합니다.

API 계약과 화면 흐름 동시 확인

요청/응답 필드, 에러 포맷, 인증 필요 여부를 먼저 맞추고 실제 화면에서 사용자가 거치는 흐름까지 확인합니다.

후속으로 남긴 판단

단일 서버 기준으로 충분했던 scheduler guard는 멀티 서버 환경에서는 분산 락이 필요합니다. 게임 가이드 탭 조회는 현재 구조에서 화면 요구를 만족하지만, 데이터가 커질 경우 Repository Pageable 조회로 옮겨 DB에서 필요한 페이지 단위만 가져오는 개선이 필요합니다.
