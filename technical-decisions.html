Decision Records

주요 기술적 의사결정

아래 항목은 “무엇을 썼다”보다 “왜 그렇게 나눴고, 어떤 위험을 줄였는가”에 초점을 둔 기록입니다.

Auth

JWT + Redis 인증 흐름을 서버가 통제할 수 있게 설계

배경

일반 로그인, Google OAuth2, 로그아웃, 토큰 재발급이 같은 인증 경계를 공유해야 했습니다.

결정

Access Token은 요청 인증에 사용하고 Refresh Token은 Redis에 저장했습니다. 재발급은 Rotation으로 관리하고, OAuth 결과는 HttpOnly Cookie로 전환했습니다.

이유

로그아웃과 재발급을 서버에서 통제할 수 있고, 토큰이 URL이나 응답 body에 직접 노출되는 위험을 줄일 수 있었습니다.

검증

Redis 저장값과 TTL, 재발급 전후 토큰 불일치, OAuth 1회용 코드 교환, HttpOnly Cookie 동작을 PR 캡처와 Swagger 흐름으로 확인했습니다.

API Contract

DTO와 Swagger 문서를 API 계약으로 관리

배경

여러 팀원이 동시에 도메인을 개발하면서 요청/응답 예시와 실제 DTO가 어긋나는 문제가 생겼습니다.

결정

Controller 문서화 클래스를 분리하고 Request/Response DTO 네이밍, responseCode 예시, 입력 검증 기준을 맞췄습니다.

이유

Swagger가 프론트엔드와 QA의 확인 기준이기 때문에 문서 불일치 자체가 기능 결함으로 이어질 수 있었습니다.

검증

회원가입 검증, 비밀번호 정책, birthDate 검증 오류를 이슈 단위로 재현하고 Swagger 응답과 테스트 결과를 남겼습니다.

Payment

결제 검증은 클라이언트가 아니라 서버 기준으로 처리

배경

유료 기능 진입 전에 결제 성공 여부와 구독 활성 상태를 신뢰할 수 있는 기준으로 확인해야 했습니다.

결정

PortOne V1 API의 imp_uid 기반 조회와 금액 검증을 서버에서 수행하고 payment/subscription 엔티티로 구독 상태를 관리했습니다.

이유

클라이언트 결과만 신뢰하지 않고 외부 결제 API의 실제 응답과 서버 상태를 맞춰야 결제 우회 가능성을 줄일 수 있었습니다.

검증

결제 조회, 금액 불일치, 구독 취소, 관리자 결제내역 조회 조건을 PR 본문과 Swagger 응답으로 확인했습니다.

Game Data

게임 가이드 데이터는 패치 버전 기준으로 고정

배경

챔피언, 아이템, 시너지, 증강체 정보는 패치 버전에 따라 의미가 바뀌므로 화면과 API가 같은 기준을 바라봐야 했습니다.

결정

CDragon 데이터를 수집하고 게임 가이드와 패치노트 화면에서 패치 버전 기준 조회가 흔들리지 않도록 API 계약을 먼저 정리했습니다.

이유

외부 데이터가 갱신되는 시점과 사용자가 조회하는 시점을 분리하지 않으면 최신 데이터와 화면 설명이 어긋날 수 있었습니다.

검증

스케줄러, 수집 결과, API 응답 필드, 패치 필터 동작을 PR과 화면 캡처로 확인했습니다.

Concurrency

패치와 가이드 갱신 순서를 하나의 공용 락으로 보호

배경

패치노트와 게임가이드가 서로 다른 named lock을 사용하면 각 작업의 중복은 막아도, 다중 인스턴스에서 가이드가 패치 커밋 전의 이전 버전을 읽는 실행 순서 경쟁은 남았습니다.

결정

패치→가이드 전체 구간을 공용 MySQL 락(GET_LOCK)으로 보호하고, 저장이 끝난 패치 버전을 가이드 작업에 직접 전달했습니다. 자동 스케줄러와 관리자 수동 수집도 같은 보호 로직을 사용합니다.

실패 경계

패치 실패·빈 버전이면 가이드를 실행하지 않고, 가이드 실패 시에는 이미 성공한 패치를 되돌리지 않은 채 기존 ACTIVE 가이드를 유지하고 다음 실행에서 재시도합니다. 경합 중 수동 요청은 HTTP 409로 구분했습니다.

검증

실제 MySQL 8 두 세션에서 GET_LOCK=1/0, RELEASE_LOCK=1을 확인하고 락 경합·패치 실패·버전 누락·해제 실패를 회귀 테스트로 고정했습니다. 최종 반영은 PR #773에서 확인했습니다.

Data Safety

외부 데이터는 DB 반영 전에 완전성을 검증

배경

외부 페이지가 정상 응답해도 구조 변경이나 파서 상한으로 일부만 추출되면, 새 결과에 없는 기존 자동 수집 데이터를 삭제할 수 있었습니다.

결정

빈 결과·치명적 파서 경고·필수값 누락·중복 수집 식별자를 거부하고, 기존 데이터 유지율이 50% 미만이면 DB 변경 전에 수집을 중단했습니다. 수동 편집 행은 계산과 삭제 대상에서 제외했습니다.

이유

요청과 파싱의 성공만으로 데이터가 안전하다고 볼 수 없습니다. 기존 데이터를 지우기 전에는 실제 삭제 기준과 같은 식별자로 완전성을 확인해야 했습니다.

검증

행 수는 같지만 수집 식별자의 교집합이 0인 반례까지 검증했습니다. 구현과 상세 테스트 근거는 PR #753에 남겼습니다.

AI

AI 입력과 출력을 각 서버에서 나눠 검증

배경

GameGuide AI Pathfinder는 프론트 채팅 UI, Spring API, FastAPI AI 서버, OpenAI 응답이 함께 맞물리는 기능이었습니다.

결정

React는 선택 화면, Spring은 인증·사용자별 호출 제한·DB 기반 가이드 키 검증, FastAPI는 프로세스별 호출 제한·OpenAI 연동·응답 형식 검증을 담당하도록 나눴습니다.

이유

익명·반복 요청의 비용 노출을 제한하고, 클라이언트 입력과 LLM 출력 모두를 신뢰하지 않는 값으로 취급하면서 기존 API 계약과 화면 상태를 유지하기 위해서였습니다.

검증

가이드 근거와 AI 제안을 분리하고, 모델이 반환한 단계별·추천·출처 키를 서버의 허용 목록으로 필터링했습니다. 프롬프트 공격·지원하지 않는 통계 요청·JSON 파싱 실패·OpenAI 장애를 테스트하고 프론트·Spring·AI 서버 CI를 통과했습니다.

남은 한계

Spring·FastAPI의 호출 제한과 장애 차단 상태는 프로세스 메모리에 저장됩니다. 다중 인스턴스로 확장할 때는 Redis 같은 공용 저장소로 옮겨야 합니다.

Performance

성능 개선은 체감 추정보다 측정 결과 기준으로 분리

배경

게임가이드 첫 진입과 패치노트 changes 조회가 상대적으로 무거운 API로 확인됐습니다.

결정

게임가이드 탭 조회는 직접 DB Pageable 방식으로 전환했고, 팀원이 구현한 패치 버전 선조회 경량화는 코드와 측정 결과를 검증했습니다. 패치노트 통계는 여러 건수 조회를 단일 집계 쿼리로 직접 통합했습니다.

이유

성능 문제를 전체 구조 탓으로 넓히기 전에 어느 API가 무거운지, 왜 무거운지 먼저 나눠야 했습니다.

대안 비교

통계 계산을 프론트로 옮기는 대안도 JMeter로 케이스별 1회 근사했습니다. 실제 브라우저 실험이 아니므로 구조 변경의 근거로 일반화하지 않았고, 확인된 DB 연결 대기는 후속 병목 후보로 분리했습니다.

검증

로컬 백엔드와 MySQL 8에 실제 TFT 17.6 데이터를 넣고, 50 VU·300초 조건으로 변경 전후를 각각 3회 비교했습니다. 4개 API 혼합 시나리오에서 TPS 11.1% 향상, p99 21.7% 감소, 오류율 0%를 확인했습니다. 이 중 필터 2개는 당시 데이터에서 0건 응답이었습니다.

Release

운영 배포 설정은 로컬 QA 설정과 분리

배경

배포 환경에서는 CORS, OAuth 리디렉션, AI 서버 URL, OpenAI 키, DB 마이그레이션, 상태 확인, ALB 라우팅처럼 실패 지점이 많았습니다.

결정

환경 변수와 비밀값 파일을 분리하고 Docker/Flyway 기본값에 로컬 테스트 초기 데이터가 섞이지 않도록 운영 경로를 정리했습니다.

이유

로컬 검증용 데이터나 관리자 계정이 운영 DB에 들어가는 위험은 기능 오류보다 복구 비용이 큰 운영 사고로 이어질 수 있었습니다.

검증

배포 전 체크리스트로 주요 화면, 게임 가이드 조회, 패치노트 조회, AI Pathfinder 요청, 로그와 환경 변수를 순서대로 확인했습니다.

Verification Evidence

검증을 남긴 방식

PR / Issue

기능 구현 의도, 변경 범위, 재현 조건, 확인 결과를 이슈와 PR에 연결했습니다. 리뷰 코멘트가 생기면 수정 커밋으로 반영해 논의가 코드에 어떻게 반영됐는지 추적했습니다.

Swagger / API Contract

요청/응답 필드, 상태 코드, 입력 검증, 예시 응답을 Swagger 기준으로 맞췄습니다. 문서가 실제 DTO와 어긋나면 기능 결함으로 보고 수정했습니다.

Test / Smoke

단위 테스트, 백엔드 전체 테스트, Docker MySQL과 백엔드 연동 확인, 상태 확인 API, 실제 화면 접근으로 배포 전후 핵심 흐름을 검증했습니다.

Load / Metrics

같은 로컬 조건에서 변경 전후를 반복 측정하고 평균·상위·최대 응답시간을 함께 비교했습니다. 별도 1,000 VU 근사 실험은 성과 수치가 아니라 DB 연결 대기라는 다음 병목을 찾는 데만 사용했습니다.

AI Guard

프롬프트 공격, 가이드 키 변조, 없는 통계 생성 요청을 입력하고 위험 문구 필터, 허용 키 보정, 대체 응답을 확인했습니다.

Operating Principles

앞으로도 유지할 기준

이 섹션은 “현재 구현이 완성형이다”라는 의미가 아니라, 다음 프로젝트에서도 반복해서 지키려는 작업 원칙입니다. 어떤 기술을 새로 쓰더라도 API 계약, 보안 경계, 데이터 갱신, 배포 설정, 검증 기록을 같은 기준으로 확인하겠다는 뜻입니다.

rule

계약을 먼저 맞춘다

화면 구현보다 요청/응답 필드, 상태 코드, 에러 포맷, 인증 필요 여부를 먼저 정리합니다.

shield

중요한 값은 서버에서 다시 확인한다

인증, 결제, AI 응답 검증처럼 악용 가능성이 있는 흐름은 클라이언트 결과만 믿지 않습니다.

sync

외부 데이터는 반복 실행을 고려한다

데이터 수집, 스케줄러, 마이그레이션은 같은 작업이 다시 실행돼도 중복이나 운영 오염이 생기지 않도록 기준을 둡니다.

settings

환경 설정을 분리한다

로컬·개발·운영 환경의 변수, 초기 데이터, 비밀값, 리디렉션 URL을 분리해 배포 기본값이 안전하도록 둡니다.

speed

성능은 측정 후 고친다

느릴 것 같은 지점이 아니라 실제 수치가 튀는 API를 기준으로 원인과 후속 개선을 분리합니다.

fact_check

검증 결과를 기록한다

테스트 통과 여부, Swagger 캡처, 배포 확인 결과와 후속 점검 항목을 PR과 이슈에 남깁니다.

남은 과제

AI 호출 제한과 장애 차단 상태를 다중 인스턴스가 공유하도록 Redis 등으로 옮기고, 고부하 근사 실험에서 확인한 DB 연결 대기를 실행계획·인덱스·연결 풀 설정 기준으로 추가 검증해야 합니다.
