시스템 구조

서비스 구조와 사용자 흐름

팀 전체 요청 흐름과 역할별 기능을 먼저 보여준 뒤, 제가 맡은 게임가이드·패치노트 백엔드와 AI 연동·배포 검증 범위를 구분해 설명합니다.

ClientReact 사용자 화면가이드·패치·AI 요청
RoutingALB · Nginx정적 화면과 API/OAuth 분기
본인 담당 중심Spring BootGuide · Patchnote · AI 계약 검증
Data pathMySQL · CDragon패치 기준 수집·조회·공용 락
AI pathFastAPI · OpenAI허용 목록·fallback·circuit breaker

데이터 경로와 AI 경로는 Spring Boot에서 각각 분기됩니다. 도식은 현재 main 배포 워크플로와 담당 범위를 기준으로 단순화했습니다.

01 / 배포 근거

AWS ECS

GitHub Actions에서 ECS까지 이어지는 배포 흐름

현재 main 배포 워크플로 기준으로 프론트엔드·Spring Boot·FastAPI가 각 Docker 이미지를 Amazon ECR에 푸시하고 ECS 태스크 정의와 서비스를 갱신합니다. 프론트 컨테이너의 Nginx는 정적 화면을 제공하고 ALB가 API·OAuth 경로를 백엔드로 분리합니다.

이미지 빌드

프론트엔드·백엔드·AI 서버를 서비스별 Docker 이미지로 만들고 ECR 저장소에 푸시합니다.

서비스 갱신

새 이미지 태그를 ECS 태스크 정의에 반영하고 각 워크플로에서 해당 ECS 서비스를 갱신합니다.

안정 상태 확인

Actions는 ECS 안정 상태까지 대기합니다. 실제 반영은 실행 중 task revision·image SHA, health와 공개 API smoke를 별도로 확인합니다.

02 / 사용자 흐름

사용자와 관리자의 기능 범위를 역할별로 정의

일반 사용자는 인증, 전적·메타 탐색, 게임 가이드, 패치노트, 커뮤니티, AI 추천을 이용합니다. 관리자는 메타·영웅 증강·게임 가이드·패치노트 데이터를 관리하고 외부 데이터 수집과 운영 모니터링을 수행합니다. 이 중 게임가이드와 패치노트 도메인을 담당했고, 해당 데이터를 사용하는 GameGuide AI Pathfinder 연동과 관련 운영 안정화에 기여했습니다.

사용자 경험

팀 서비스는 로그인부터 전적과 메타 탐색, 가이드 조회, 패치 변화 확인, 커뮤니티 참여, AI 추천까지 하나의 탐색 흐름을 제공합니다.

운영자 도구

데이터 수집과 CRUD, JSON 검증, 캐시 통계와 Riot API 호출 제한 확인 등 서비스 운영 기능을 별도 권한으로 구분했습니다.

Architecture Notes

구현하면서 고려한 기술 포인트

게임 데이터와 패치노트 수집, AI 서버 연동, 운영 배포 설정은 서로 다른 실패 지점을 갖습니다. 그래서 기준 데이터를 먼저 고정하고 실패 시 원인을 추적할 수 있는 구조를 우선했습니다.

API 계약

프론트엔드와 맞춰야 하는 요청/응답, 검색 조건, 페이지네이션, 실패 응답을 먼저 정리했습니다. 게임 가이드와 패치노트 화면이 사용하는 필드는 패치 버전 기준이 흔들리지 않도록 나눴습니다.

데이터 수집 기준

0건·잘림·필수값 누락·원본 식별자 중복을 DB 반영 전에 차단했습니다. 기존 자동수집 항목의 50% 미만만 유지되면 작업을 중단해 마지막 정상 데이터를 보존했습니다.

초기 단일 서버 보호 로직을 MySQL 공용 락으로 확장해 여러 인스턴스의 갱신 경쟁을 차단하고, 패치가 저장한 버전을 가이드 작업에 명시적으로 전달했습니다. 가이드 생성이 실패하면 기존 활성 가이드를 유지해 다음 실행에서 재시도하며, 자동·관리자 수동 실행은 같은 보호 경로를 사용합니다.

AI 연동

Spring에서 로그인·사용자별 호출 횟수·요청한 가이드 키를 검증했습니다. 모델 응답 키는 서버 허용 목록과 다시 대조했고, Spring의 401·429 오류와 AI 서버 장애·파싱 실패 시 결정적 fallback을 구분했습니다.

운영 확인

배포 후 로그, 환경 변수, AI 서버 연결, 컨테이너 프로필, 헬스체크와 공개 API 응답을 점검했습니다. 오류가 발생했을 때 어느 구간에서 실패했는지 추적할 수 있도록 체크리스트를 남겼습니다.

협업과 릴리즈

팀 협업과 릴리즈 관리

TFT-gogo는 여러 사람이 동시에 작업하는 팀 프로젝트로 진행했습니다. 도메인 명세, API 계약, 이슈와 PR, 배포 체크리스트, 릴리즈 이후 확인 기준을 연결해 기능 구현이 실제 실행 가능한 상태까지 이어지도록 관리했습니다.

01 / 요구사항

도메인 기준과 역할을 먼저 맞췄습니다

게임 가이드, 패치노트, AI 추천, 커뮤니티처럼 도메인이 나뉘어 있어 담당 범위와 API 필요 항목을 문서로 정리했습니다. 변경된 요구사항은 구현 전에 문서에 먼저 반영해 팀원이 같은 기준을 보고 작업하도록 맞췄습니다.

  • 관리 항목 도메인 명세, 화면 정의, 역할 분담, 완료 기준
  • 확인 방식 작업 전 문서 기준을 맞추고 이슈로 구현 범위를 분리

02 / Pull Request

Issue, Branch, PR을 연결해 변경 이력을 남겼습니다

GitHub Issue를 기준으로 브랜치를 만들고 PR에서 구현 의도, 변경 범위, 확인 방법을 공유했습니다. 리뷰 코멘트가 생기면 수정 커밋으로 반영해 논의가 코드에 어떻게 반영됐는지 추적할 수 있게 했습니다.

  • 관리 항목 브랜치 전략, 커밋 컨벤션, PR 설명, 코드리뷰 코멘트
  • 확인 방식 병합 전 변경 파일과 테스트 결과를 PR에 함께 기록

03 / 릴리즈

API 계약부터 배포 후 검증까지 확인했습니다

요청/응답과 인증 조건을 먼저 맞추고 환경 변수, DB·AI 서버 연결, task revision과 image SHA, health, 공개 API 응답을 순서대로 확인했습니다. Actions의 안정화 성공만으로 실제 반영을 단정하지 않았습니다.

  • 관리 항목 API 계약, 환경 변수, DB 연결, AI 서버 URL, OpenAI API 키, 빌드 결과
  • 확인 방식 배포 전 체크리스트와 배포 후 공개 smoke를 분리해 기록
